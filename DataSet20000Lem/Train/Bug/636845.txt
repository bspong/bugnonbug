linux 64 bit build do not properly load system plugins or extension
Created attachment 515193
this patch set the location of the system plugins and extension folder based on being a 64bit o instead of the libdir setting

I have noticed that while running nighlties that system installed plugins (i.e. plugins installed through apt/yum) aren't detected by minefield.

Using strace I found that these build are actually scanning /usr/lib/mozilla/plugins instead of /usr/lib64/mozilla/plugins on linux64. The way that the code currently figure out whether to use /usr/lib64 or /usr/lib is to search for 'lib64' in the autoconf libdir [1]. Instead, I think it should be set to a commonly used value that work on the majority of distributions.

The patch I have attached change the system plugin path from being based on the libdir autoconf.mk value to choosing the path based on whether the system is a 64bit Linux OS. I have verified that the /usr/lib64/mozilla/plugins is the default for CentOS [2] and Fedora [3] and /usr/lib64 is a symlink for /usr/lib on Debian[4] and Ubuntu [5] which make this change invisible.

I have also verified that this patch fix the behaviour on my system. I have made the same change for the system extension directory which used the same method of figuring out whether or not to use /usr/lib64.

With my patch applied I have verified that my build is finding my system plugins and extensions.

ac_add_options --prefix=${PREFIX}
ac_add_options --libdir=${PREFIX}/lib64

but doe not address the root problem.
